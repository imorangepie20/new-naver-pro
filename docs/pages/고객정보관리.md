# 고객정보관리

## 페이지 목적

- 매물 기준이 아니라 고객 기준으로 거래 이력과 계약 현황을 한 화면에서 조회하는 페이지
- 같은 고객이 여러 매물과 여러 계약을 가지고 있을 때, 고객별로 묶어서 빠르게 확인하는 것이 목적
- 전화, 재계약, 후속 연락, 메모 확인 같은 실무 작업을 고객 단위로 처리할 수 있어야 함

## 필요한 이유

- 현재 관리매물 화면은 `매물 중심`이라 같은 고객의 여러 계약을 한 번에 보기 어려움
- 고객명이나 연락처로 업무를 시작하는 경우가 많아 고객 기준 화면이 별도로 필요함
- 계약 만기 예정 고객, 연락처 누락 고객, 최근 메모가 있는 고객을 빠르게 찾는 용도가 큼

## 1차 구현 범위

- 별도 고객 마스터 테이블 없이 `ManagedProperty` 기반으로 고객 목록을 구성
- 화면 성격은 `고객 통합 조회 + 고객 단위 후속관리`에 집중
- 1차에서는 고객 단독 등록/삭제 기능 없이, 계약 데이터에서 고객 정보를 묶어 보여주는 방식으로 시작
- 데이터 소스는 우선 `관리매물` 기준으로 한정
- 향후 필요 시 `관심매물`, `업로드 매물`, 별도 고객 마스터로 확장 가능

## 화면 구성

### 1. 상단 요약 영역

- 총 고객 수
- 진행중 계약 고객 수
- 30일 이내 만기 예정 고객 수
- 연락처 누락 고객 수

### 2. 검색 / 필터 영역

- 통합 검색

  - 매물명
  - 건물명
  - 주소
  - 메모
- 상태 필터
  - 전체
  - 진행중
  - 만료
  - 갱신
- 거래유형 필터
  - 전체
  - 매매
  - 전세
  - 월세
- 만기일 필터
  - 전체
  - 7일 이내
  - 30일 이내
  - 90일 이내
  - 만기 지남
- 연락처 필터
  - 전체
  - 전화번호 있음
  - 전화번호 없음
- 정렬
  - 최근 계약일 순
  - 만기일 임박 순
  - 고객명 순
  - 최근 수정일 순

### 3. 고객 그리드 영역

- 고객 1명당 상위 행 1개
- 상위 행 펼치면 해당 고객의 계약/매물 이력을 하위 행으로 표시
- 기본은 상위 행 접힘 상태
- 검색 결과가 1건일 때는 자동 펼침 허용

## 상세 동작

### 고객 묶음 기준

- 동일 고객 판단 기준은 `고객명 + 전화번호` 조합을 우선 사용
- 전화번호는 하이픈 제거 후 비교
- 이름만 같고 전화번호가 다르면 다른 고객으로 분리
- 전화번호가 없으면 이름만으로 자동 병합하지 않고 `확인 필요` 상태로 취급하는 것이 안전

### 만기일 강조

- 30일 이내 만기 예정은 강조 표시
- 7일 이내 만기 예정은 더 강한 강조 표시
- 이미 만기일이 지난 건은 별도 색상 또는 배지 처리

### 메모 표시 규칙

- 상위 고객 행에는 가장 최근 메모 1줄만 요약 표시
- 하위 계약 행에는 해당 계약의 메모 원문 표시
- 메모가 길면 말줄임 처리하고 hover 또는 상세 보기로 전체 확인

### 연락처 처리

- 전화번호가 있으면 클릭 또는 복사 가능
- 전화번호가 없으면 `연락처 없음` 표시
- 연락처 누락 고객은 필터링 가능해야 함

## 고객 상세 보기

- 고객 상위 행 클릭 시 우측 패널 또는 모달로 상세 정보 표시 가능
- 상세 화면에는 아래 정보 포함
  - 고객 기본 정보
  - 계약 이력 전체
  - 최근 메모
  - 만기 예정 계약
  - 알림/연락 이력

## 데이터 기준과 주의사항

### 현재 스키마 기준

- 현재 `ManagedProperty`에는 아래 필드가 이미 존재함
  - `articleName`
  - `buildingName`
  - `address`
  - `propertyType`
  - `contractType`
  - `contractDate`
  - `contractEndDate`
  - `tenantName`
  - `tenantPhone`
  - `managerName`
  - `managerPhone`
  - `notes`
  - `status`

### 현재 구조의 한계

- 현재 스키마에는 `고객`을 일반화한 공통 필드가 없음
- `tenantName`, `tenantPhone`은 전세/월세에는 맞지만 매매까지 포괄하기엔 의미가 좁음
- `managerName`, `managerPhone`은 책임자 정보라 고객 정보와 동일하지 않을 수 있음

### 권장 데이터 정리 방향

- 고객정보관리 페이지를 안정적으로 운영하려면 아래 중 하나가 필요함
- 1안: `ManagedProperty`에 `customerName`, `customerPhone` 같은 공통 고객 필드 추가
- 2안: `tenantName`, `tenantPhone`을 사실상 거래 상대방 공통 필드로 재정의
- 3안: 별도 `Customer` 마스터 테이블을 만들고 계약과 연결

## 권장 구현 순서

### 1단계

- 관리매물 기준 고객 묶음 조회 화면 먼저 구현
- 고객 단위 검색, 만기 필터, 연락처 확인, 하위 계약 펼침 기능 제공

### 2단계

- 고객 상세 패널 추가
- 최근 연락 이력, 알림 이력, 고객 메모 강화

### 3단계

- 별도 고객 마스터 도입 여부 결정
- 관심매물/업로드매물과 고객 데이터 연결 검토

## 화면 구현 시 유의사항

- 상위 고객 행과 하위 계약 행이 한눈에 구분되어야 함
- 행 수가 많아질 수 있으므로 기본은 접힘 구조가 적합
- 검색 결과가 많을 때도 고객 기준으로 빠르게 훑을 수 있어야 함
- 모바일에서는 상위 고객 카드형, 하위 계약은 접힘 상세형으로 변환하는 방식이 적합

## 이 문서 기준 결론

- 고객정보관리 페이지는 `고객 기준 통합 조회 화면`으로 정의
- 1차는 `ManagedProperty` 기반 조회형 화면으로 구현
- 다만 장기적으로는 `고객 공통 필드` 또는 `고객 마스터` 설계가 필요함
